## Purpose

How translation keys flow from `translations.js` through `LangContext` into components, and where Mandarin currently sits while EN gets polish.

## Surface

- [`LangContext.jsx`](../../src/contexts/LangContext.jsx): minimal `createContext({ t: {} })` plus `useLang()` hook.
- [`translations.js`](../../src/translations.js): exports `T = { en, zh }`, each a flat object keyed by translation key. **ZH is frozen** as of 2026-04-28 — see [Adding a new string](#adding-a-new-string) below and [ZH translations frozen](../decisions/2026-04-28-zh-translations-frozen.md).
- [`App.jsx`](../../src/App.jsx) is the only Provider site:

```jsx
<LangContext.Provider value={{ t, lang }}>
  ...
</LangContext.Provider>
```

`lang` is currently hardcoded to `"en"` in `App.jsx`; `t = T.en`.

## Mandarin status (stashed)

Mandarin (`T.zh`) and `lang === "zh"` branches throughout components are intentionally preserved so that reviving Mandarin only needs:

1. Restore the `lang` state + a UI toggle (was: `localStorage` key `bite_lang`).
2. Optionally re-enable `welcome_zh_*` Supabase overrides.

See [2026-04-28 — Stash Mandarin localization](../decisions/2026-04-28-stash-mandarin-localization.md). The taste-personality engine is English-only and would need a parallel zh build (or a `lang` arg threaded through) — see [`features/taste-personality.md`](taste-personality.md).

## Conventions

- Keys are camelCase: `myLog`, `myTaste`, `communityHintGlobal`, `welcomeBtn`.
- Nested labels for tier names use the same key on both bands: `sucks`, `meh`, `average`, `goodTaste`, `great`, `elite`.
- Welcome copy can be overridden from `settings.welcome_{en|zh}_{title|body}` when `VITE_WELCOME_USE_SUPABASE=true`. See [`features/auth-rls.md`](auth-rls.md).

## Adding a new string

While ZH is frozen ([decision](../decisions/2026-04-28-zh-translations-frozen.md)):

1. Add the key under `T.en` only in [`translations.js`](../../src/translations.js). Do **not** add a matching `T.zh` entry — net-new ZH copy is deferred to a single sweep pass once English is stable.
2. Editing an existing English value? Leave the existing ZH value as-is even if it now reads stale.
3. In components, use `const { t } = useLang(); ... t.myNewKey`. ZH users hitting a missing key fall through to English (existing context behavior).
4. If the string is user-facing welcome copy, also document the optional `settings` key.

When the user lifts the freeze ("translations unfrozen" / "do the ZH pass"), backfill ZH entries for every key added or edited since 2026-04-28 in one sweep, using `git log -- src/translations.js` as the worklist.
